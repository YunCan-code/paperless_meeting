# 开发阶段会场稳定性检查单

## 目标

这份检查单不是为了在开发阶段硬凑 50 台真机，而是用更低成本的方法尽早发现会议现场最容易掉链子的 P0/P1 问题。

建议把验证拆成 3 层：

- 接口并发层：用脚本替代大批量设备，验证后端是否会被打崩
- 状态一致性层：用少量浏览器标签页和 1 到 2 台 Android，验证断线、重启、误触后的状态是否自愈
- 真实终端层：至少保留 1 台真 Android 或 1 个模拟器，验证 PDF、Socket、页面恢复这类端上问题

## 最小验证环境

- 1 套 `docker-compose` 测试环境
- 1 个后台管理端浏览器
- 1 个大屏页浏览器
- 1 到 2 个移动端浏览器标签页
- 1 台 Android 真机
- 1 台 Android 模拟器或第二台设备
- 20 到 50 个测试用户
- 1 个可重复创建和删除的测试会议
- 1 份体积偏大的 PDF

## 推荐执行顺序

1. 管理员误触两次开始/结束投票
2. 后端重启恢复
3. 断网 30 秒恢复
4. 并发提交投票
5. 并发打开 PDF
6. 并发进入会议
7. Redis 暂停
8. Android 新旧版本混用
9. 旧会议可见性规则

## 场景清单

### 1. 50 台平板同时进入同一会议

**目标**

- 验证会议列表、会议详情、互动总览在并发请求下是否稳定

**前置条件**

- 已创建测试会议
- 至少准备 1 个 `meeting_id`
- 如需模拟 Android 可见性规则，准备 1 个 `user_id`

**执行方式**

- 运行 [test-meeting-entry.ps1](../scripts/stability/test-meeting-entry.ps1)
- 同时打开后台页、大屏页、移动端页各 1 个观察首屏

**示例**

```powershell
pwsh ./scripts/stability/test-meeting-entry.ps1 `
  -BaseUrl http://127.0.0.1:8000 `
  -MeetingId 12 `
  -UserId 5 `
  -RequestCountPerEndpoint 50 `
  -Concurrency 20
```

**通过标准**

- 3 个接口都没有 500、超时、数据库连接错误
- 页面可以正常进入会议
- 平均耗时和 P95 没有明显失控

**警告标准**

- 少量 4xx 但确认由测试数据导致
- 页面可用，但接口延迟明显抬升

**失败标准**

- 出现 500、超时、连接池耗尽、页面打不开

**重点记录**

- backend 日志
- 数据库连接错误
- 接口汇总结果

### 2. 50 台同时打开同一个 PDF

**目标**

- 验证静态文件分发是否压垮后端

**前置条件**

- 已知一个可访问的 PDF 路径，例如 `/static/example.pdf`

**执行方式**

- 运行 [test-pdf-load.ps1](../scripts/stability/test-pdf-load.ps1)
- 同时手工在后台刷新一次互动总览或切换一次投票状态

**示例**

```powershell
pwsh ./scripts/stability/test-pdf-load.ps1 `
  -BaseUrl http://127.0.0.1:5000 `
  -PdfPath /static/example.pdf `
  -RequestCount 50 `
  -Concurrency 20
```

**通过标准**

- PDF 大部分请求稳定返回 `200`
- 管理端操作没有明显被拖慢
- backend 日志里不应再看到大批量 `/static/` 回源

**失败标准**

- 后端被静态文件流量挤占
- 控制端操作出现明显卡顿或超时

**重点记录**

- nginx 日志
- backend access log
- 管理端是否卡顿

### 3. 50 台在 10 秒内提交投票

**目标**

- 验证投票接口在高并发提交下是否稳定
- 验证截止前后状态与真实收票行为是否一致

**前置条件**

- 测试会议中有一个刚开始的投票
- 已知 `vote_id`
- 已知一个或多个有效 `option_id`
- 准备 20 到 50 个尚未参与该投票的测试用户

**执行方式**

- 运行 [test-vote-submit.ps1](../scripts/stability/test-vote-submit.ps1)
- 同时打开后台、大屏、移动端各 1 个观察票数变化

**示例**

```powershell
pwsh ./scripts/stability/test-vote-submit.ps1 `
  -BaseUrl http://127.0.0.1:8000 `
  -VoteId 18 `
  -OptionIds 33 `
  -UserCsvPath ./scripts/stability/sample-users.csv `
  -Concurrency 20
```

**通过标准**

- 投票开始后可以正常收票
- 截止后全部拒绝，不再混入迟到票
- 后台、大屏、移动端票数一致

**失败标准**

- 界面显示可投但大量提交失败
- 投票已结束后仍能收票
- 总票数和提交成功数对不上

**重点记录**

- 成功数与失败数
- 首批失败响应体
- 大屏和后台的最终总票数

### 4. 管理员误触两次“开始投票”

**目标**

- 验证开始/结束投票是否有幂等保护

**前置条件**

- 准备 1 个草稿投票
- 同时打开两个后台标签页

**执行步骤**

1. 两个标签页几乎同时点击开始投票
2. 观察倒计时是否被重置
3. 投票进行中再重复点击结束投票
4. 观察是否出现第二次真实切换

**通过标准**

- 只发生一次真实开始
- `started_at` 不被后一个请求覆盖
- 只发生一次真实结束

**失败标准**

- 倒计时跳回去
- 状态切换重复广播
- 后台出现两次成功但状态不一致

### 5. 一半设备断网 30 秒后恢复

**目标**

- 验证 Socket 断线重连后是否能自动补拉当前状态

**前置条件**

- 后台页、大屏页、移动端页、Android 真机/模拟器均已进入同一会议

**执行步骤**

1. 让其中一半页面和设备断网 30 秒
2. 断网期间开始或结束一次投票
3. 恢复网络，不手工刷新页面
4. 观察是否自动回到当前状态

**通过标准**

- 重连后自动补拉
- 不需要人工下拉或刷新
- 大屏、Web、Android 最终状态一致

**失败标准**

- 某一端停留在旧页面或旧倒计时
- 必须人工刷新才能恢复

### 6. 后端重启一次

**目标**

- 验证服务重启后，会议中的状态能否从数据库恢复并推回客户端

**执行步骤**

1. 在投票进行中重启 backend 容器
2. 页面保持不刷新
3. 等待服务恢复
4. 观察客户端是否自动重连和补拉

**通过标准**

- 页面自动恢复连接
- 状态从数据库恢复
- 当前投票、互动总览与实际状态一致

**失败标准**

- 页面永久停留在旧状态
- 投票状态与数据库脱节

### 7. Redis 暂时不可用

**目标**

- 验证多 worker 下 Redis 异常时是否会出现实时广播分裂

**前置条件**

- 继续使用 4 worker 运行 backend

**执行步骤**

1. 暂停 Redis
2. 在后台发起开始投票或结束投票
3. 观察不同页面是否都收到相同结果
4. 记录 backend 日志中的 Redis 和 Socket.IO 错误

**通过标准**

- 不建议把此项定义为“可接受降级”
- 只要出现状态分裂，就视为阻塞项

**失败标准**

- 不同页面状态不一致
- 同一会议内部分页面收到了广播、部分没有

### 8. Android 新旧版本混用

**目标**

- 验证旧版 APK 与新版 APK 混用时是否会出现状态恢复、投票提交行为不一致

**前置条件**

- 至少保留 1 台旧版 APK 和 1 台新版 APK

**执行步骤**

1. 新旧版同时进入同一会议
2. 一起参与投票
3. 分别做断网恢复和后端重启恢复
4. 对比两端表现

**通过标准**

- 新旧版在核心链路上表现一致

**失败标准**

- 旧版无法自动补拉
- 新旧版在投票临界点行为不同

### 9. 会议结束后旧会议是否还能被误进入

**目标**

- 验证 Android 会议可见性规则是否按当前实现生效

**前置条件**

- 创建一个已结束的测试会议
- 准备未签到用户和已签到用户各 1 个
- 明确 `meeting_visibility_hide_after_hours`、会议自身 `android_visibility_mode`

**执行步骤**

1. 用未签到用户访问会议列表和会议详情
2. 用已签到用户访问同一会议
3. 分别测试 `inherit`、`custom_hours`、`hidden`、`always_show`

**通过标准**

- 未签到用户遵守当前可见性窗口
- 已签到用户仍按规则继续可见
- 不可见会议详情返回 `404`

**失败标准**

- 旧会议被未授权用户继续进入
- 可见性与配置不一致

## 每次演练必须留痕

建议每次测试都复制 [result-template.md](../scripts/stability/result-template.md) 作为记录，至少保留这些信息：

- 测试时间
- 测试会议 ID
- 测试页面和设备数量
- 脚本参数或手工步骤
- 结果：通过 / 警告 / 失败
- 失败截图
- backend / nginx / redis 关键日志

## 结论口径

开发阶段不需要证明“50 台真机完全等价复刻成功”，而是要尽快回答下面 4 个问题：

- 并发请求会不会把后端打崩
- 状态在断线和重启后会不会分裂
- 控制端误触会不会造成二次开始或二次结束
- 旧会议和旧版本会不会绕过当前规则误进入

如果前 4 项检查稳定通过，已经能覆盖大多数“会议现场会不会掉链子”的核心风险。
